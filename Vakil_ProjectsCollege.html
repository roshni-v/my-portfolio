Museum App Project

Course: CSC 308 & 309 (Software Engineering I & II)

Duration: Approximately 2 Months (10/19/22 - 12/7/22)

Objective: To Develop a Multi-Platform Application (Web, iOS, Android) For the Purpose of Advertising Local Museum Events and Allowing Users to View and Register for These Events

Type of Project: Group Project

The purpose of this project was to go through the software development life-cycle to create a functioning .NET MAUI Blazor application fulfilling the criteria in the objective stated above. Documentation for the initial parts of this process (writing user stories, creating functional requirements, designing UI prototypes using Figma, etc) can be found here. I collaborated with my group on all of these processes, and was exclusively responsible for creating the “add new event” activity, class, and sequence diagrams found in the aforementioned documentation. Once we began coding, I focused primarily on developing the “Ticket Details” page of the application, as well as all of the data collection logic that went behind it (found in TicketDetails.razor). This necessitated familiarizing myself with the basics of C# and XAML (since that is what the MAUI framework utilizes), as well as the Radzen Blazor component library (since that is what we decided to use in order to streamline our UI and make it look more professional, as opposed to doing all styling primitively with CSS). My greatest challenges were figuring out 1) how to grab the information for each person inputted by the user, 2) how and when to populate new and distinct Person objects with this inputted data, and 3) how to populate new fields tied to a new Person object seamlessly whenever the user clicks on the “Add” button. During this development process, I researched and discovered how to implement code that mandates that text fields not be empty when a button is clicked, so I incorporated this feature in other pages of our application like our “Payment” page (Payment.razor) and “Sign Up” page (SignUp.razor). Ultimately, I was definitely not my group’s strongest or most experienced developer, but I learned a lot during this project and contributed well and to the best of my abilities. I am a strong collaborator, whether I am in a position of technical strength and can therefore lead, or in a position where I am less experienced and follow the lead of others. As a result, I earned 100% on both peer evaluations associated with this project. At the end of the quarter, my group presented our final product and development process to our class. We earned an A on this project. Our final codebase can be found here.

Car Racing Game—Final CPE 123 Project

Our final game (can be played on a laptop or desktop)

Course: CPE 123 (Introduction to Computing)

Duration: 4 Weeks (11/2/20 - 11/30/20)

Objective: To Create a Fully Functional Basic Game/Animated Short/Interactive Sketch

Type of Project: Group Project

For this final CPE 123 project, we were assigned to collaborate in a group of four to create a short animated scene, a small game, or some sort of interactive sketch (details about the project requirements can be found here). My group decided to create a game together. I came up with the basic concept of the game early on: a car racing game in which the cars are propelled forward by repeatedly pressing a certain key. It was based on my very first ever coding project (which can be found at the bottom of my high school projects section): a snail race game. My groupmates bought into the concept and added on their own requirements for the game. We decided that a main menu would display first, allowing the user to select from an array of cars. Then the race would take place. Turbos for the cars would be included. I led our meetings by setting up times and dates for Zoom calls outside of class, and then making sure we stayed on task and addressed pending issues methodically during these meetings. I delegated tasks so that everyone had something to work on each week before we came back to report the next meeting. The parts of the game that I was in charge of were: tying the movement of the cars to certain keys, varying the speeds and turbo frequencies of the different cars, programming in the speed reduction if the turbos were pressed out of time, and designing the side profiles of each car in the selection menu. The code for the finished game can be found here. Our professor commented that our project had nice physics and detail. Our thorough work earned us 104%.

Basic Hierarchical Animation Lab

My final frog-jumping animation (can be run on a laptop or desktop)

Course: CPE 123 (Introduction to Computing)

Duration: Approximately 1.5 Weeks (11/9/20 - 11/20/20)

Objective: To Animate an Articulated Character Using Hierarchical Modeling

Type of Project: Individual Project

The objective of this CPE 123 project was to primitively animate a hierarchically modeled character that we either designed ourselves or completed given the professor’s starter code. The instructions can be found here. I chose to create my own creature—a frog that would jump up and down. To begin, I created the body of the frog and then progressively built the limbs onto it. Then, I employed some trigonometry to calculate how the angles of the quads and calves segments needed to increase or decrease with the rising and falling of the frog’s body to convincingly simulate jumping. Explanations of my mathematical calculations can be found here. After the full motion was established, I incorporated a clickable region to be able to start and stop the clock controlling the jumping motion whenever the user desires. Finally, I worked in the background to make the whole animation more cohesive. This was my final code. For going above and beyond, I earned a 104% on this lab.
